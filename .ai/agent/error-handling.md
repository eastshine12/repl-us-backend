# 에러 처리 규칙

## 반드시. 에러는 코드 타입 레벨로 관리한다

에러는 문자열 상수나 임의 메시지 조합으로 흩어지게 만들지 않는다.
`ErrorCode`, `ErrorType`처럼 코드 레벨에서 관리 가능한 타입으로 정의하고, 에러의 분류와 기본 메시지를 함께 다룬다.
새로운 에러를 추가할 때도 예외 클래스부터 늘리지 말고 먼저 에러 코드 체계에 편입한다.
HTTP 상태 코드와 로그 레벨은 도메인 규칙이 아니라 전달/운영 관심사이므로 `interfaces`와 로깅 경계에서 매핑한다.

## 반드시. 예외는 CoreException 단일 진입점으로 수렴시킨다

비즈니스 예외와 시스템 예외를 제각각 여러 커스텀 예외 클래스로 퍼뜨리지 않는다.
도메인과 애플리케이션에서 제어 가능한 예외는 `CoreException` 같은 단일 코어 예외 클래스로 감싸고, `ErrorType`과 `details`를 함께 전달한다.
예외의 의미는 예외 클래스 이름 수를 늘려 표현하는 것이 아니라 에러 타입과 컨텍스트로 표현한다.

## 반드시. 전역 예외 처리는 interfaces 레이어에서 일관되게 수행한다

예외 응답 포맷과 HTTP 상태 코드는 인터페이스 계층에서 일관되게 제어해야 한다.
`GlobalExceptionHandler` 같은 전역 핸들러에서 `CoreException`을 우선 처리하고, 나머지 예외는 공통 시스템 에러 응답으로 변환한다.
핸들러는 에러 코드별 상태 코드 매핑과 응답 포맷의 단일 진입점이어야 하며, 로그 레벨 적용은 로깅 규칙과 같은 분류 체계를 사용한다.

## 반드시. 에러 응답과 에러 로그는 같은 분류 체계를 사용한다

클라이언트에 내려가는 에러 응답과 서버 내부 로그가 서로 다른 언어를 쓰면 장애 분석이 어려워진다.
에러 응답에는 `ErrorType`에서 파생된 코드와 메시지를 사용하고, 로그도 같은 코드 체계를 기준으로 남긴다.
로그 레벨은 인터페이스/로깅 경계에서 에러 코드별로 매핑해 운영자가 중요도를 바로 판단할 수 있게 한다.

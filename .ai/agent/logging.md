# 로깅 규칙

## 반드시. 로그는 요청과 흐름이 추적 가능해야 한다

로그는 단순 출력이 아니라 장애 분석 도구다.
중요한 요청 시작, 외부 연동, 상태 전이, 재시도, 보상 처리, 실패 지점에는 흐름을 추적할 수 있는 로그를 남긴다.
요청 식별자, 주요 식별자, 현재 단계, 결과 상태가 함께 보여야 한다.

## 반드시. 요청과 응답 로깅은 인터페이스 진입점에서 수행한다

HTTP 요청과 응답에 대한 공통 로깅은 `LoggingFilter` 같은 인터페이스 계층 컴포넌트에서 일관되게 처리한다.
필터, 미들웨어, 인터셉터 중 어떤 기술을 쓰든 책임은 같아야 하며, 요청 URL, 메서드, 헤더, 바디, 응답 상태, 응답 바디를 추적 가능하게 남긴다.
토큰, 쿠키, 개인정보, 영상 URL처럼 민감한 값은 반드시 마스킹하거나 제외한다.
구현 위치는 프레임워크 특성에 따라 달라질 수 있지만, 비즈니스 로직 안에 요청/응답 로깅을 흩뿌리지 않는다.

## 반드시. 에러 로그는 역추적 가능해야 한다

에러가 발생했을 때는 어떤 요청에서 어떤 입력과 어떤 단계에서 실패했는지 역추적할 수 있어야 한다.
에러 로그에는 요청 컨텍스트, 핵심 식별자, 실패한 작업, 예외 원인, 후속 영향 범위를 드러낸다.
예외를 삼키거나 맥락 없는 메시지로 다시 던지지 않는다.

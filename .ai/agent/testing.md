# 테스트 규칙

## 반드시. 테스트는 단위 테스트를 기본으로 작성한다

빠르고 좁은 테스트가 가장 먼저 실패를 알려준다.
서비스, 정책, 밸리데이터, 파사드, 컴포넌트를 추가하거나 수정할 때는 단위 테스트를 기본으로 작성한다.
특히 도메인 서비스, 밸리데이터, 정책, 애플리케이션 파사드는 테스트 없이 넘기지 않는다.

## 반드시. 테스트 시나리오는 실패와 엣지 케이스를 우선한다

뻔한 성공 시나리오만으로는 실제 장애를 막기 어렵다.
정상 흐름보다 경계값, 중복 요청, 잘못된 상태 전이, 권한 없음, 대상 없음, 재처리, 동시성 충돌, 부분 실패를 먼저 검증한다.
성공 케이스는 핵심 흐름을 설명하는 최소한만 남긴다.

## 반드시. 통합 테스트는 application부터 infrastructure까지 이어지게 작성한다

계층별 단위 테스트만으로는 실제 조합에서 깨지는 문제를 모두 잡을 수 없다.
트랜잭션, 저장, 이벤트, 락, 조회 조합이 중요한 유스케이스는 `application` 진입점에서 시작해 `domain`과 `infrastructure`를 함께 통과하는 통합 테스트를 작성한다.
저장소 구현 차이, 실제 매핑, 일관성 경계가 검증 대상이어야 한다.

## 권장. E2E 테스트는 테스트 컨테이너를 우선 검토한다

실제 인프라와 가까운 검증이 필요한 경우 인메모리 대체보다 테스트 컨테이너 기반 E2E 테스트를 우선 검토한다.
데이터베이스, 캐시, 메시지 브로커, 락, 트랜잭션 동작 차이가 중요한 경우 특히 그렇다.
운영 환경과 너무 다른 테스트 대역으로 핵심 플로우를 검증했다고 판단하지 않는다.

## 반드시. 테스트 본문은 given, when, then 주석으로 구분한다

테스트 메서드 본문은 반드시 `// given`, `// when`, `// then` 주석으로 구간을 나눈다.
이 규칙은 문서 작성 규칙이 아니라 테스트 코드 작성 규칙이다.
준비, 실행, 검증의 경계가 한눈에 보여야 하며, 준비 코드가 길어지면 픽스처나 빌더로 의도를 드러낸다.

## 반드시. 테스트 제목은 한글로 작성한다

테스트 제목은 한글로 작성한다.
무엇을 검증하는지와 어떤 상황에서 실패하거나 성공하는지가 바로 드러나야 한다.
짧더라도 의도가 분명해야 하며, 특히 실패 케이스와 엣지 케이스는 조건과 기대 결과가 제목에 드러나야 한다.

예시:

```kotlin
@Test
fun `방 정원이 가득 차면 초대 참여가 실패한다`() {
    // given
    val command = createJoinRoomCommand()
    val fullRoom = createRoom(activeMemberCount = 6)
    val facade = JoinRoomFacade(
        roomReader = FakeRoomReader(fullRoom),
        roomMemberWriter = FakeRoomMemberWriter(),
    )

    // when
    val action = { facade.join(command) }

    // then
    assertThatThrownBy { action() }
        .isInstanceOf(CoreException::class.java)
}
```

## 권장. 부하 테스트 스크립트를 운영 전 검증 자산으로 관리한다

성능이 중요한 유스케이스는 K6 같은 부하 테스트 스크립트를 작성해 재사용 가능한 검증 자산으로 남긴다.
단순 평균 응답 시간만 보지 말고 임계 상황에서의 실패율, 재시도, 대기열, 락 경합, 병목 구간을 함께 본다.
핵심 유스케이스의 성능 검증은 일회성 실험으로 끝내지 않는다.

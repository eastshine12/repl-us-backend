# 운영 규칙

## 반드시. 백엔드 public repository 기준을 지킨다

백엔드는 포트폴리오용 public repository로 공개될 수 있게 작성한다.
프론트엔드는 별도 private repository로 남는 전제를 유지하고, 백엔드 public repository에 프론트 구현 코드, private fixture, 민감한 디자인/운영 자료를 포함하지 않는다.
원격 repository 이름은 서비스명 `repl.us`를 반영해 `repl-us-backend`를 기본으로 사용하고, 호스팅 환경에서 점을 안전하게 쓸 수 있으면 `repl.us-backend`도 허용한다.
public repository에 secret, token, 개인 계정 정보, 실제 운영 endpoint, private key, `.env` 값을 커밋하지 않는다.

## 반드시. README.md는 영어로 실서비스 문서처럼 작성한다

백엔드 `README.md`는 영어로 작성한다.
README에는 프로젝트 개요, 핵심 기능, 아키텍처 방향, 기술 스택, 로컬 실행 방법, 테스트 방법, 환경 변수 예시, API 문서 위치, public repository에서 확인 가능한 개발 의도를 포함한다.
README는 단순 과제 설명이 아니라 실제 서비스 백엔드 repository를 처음 보는 사람이 실행과 구조 파악을 할 수 있는 수준으로 유지한다.

## 반드시. 코드 수정 후 린트와 규칙 적합성을 재검토한다

에이전트를 통해 코드를 수정한 뒤에는 린트, 테스트, 타입 검사 또는 그에 준하는 검증 절차를 가능한 범위에서 다시 실행한다.
검증이 끝나면 이 문서에 적힌 구조, 계층 의존성, 테스트 규칙, 책임 분리가 실제 변경사항에 반영됐는지도 다시 점검한다.
자동 검증을 통과해도 구조 원칙을 어겼다면 수정이 끝난 것이 아니다.

## 반드시. 커밋 시점이 되면 커밋을 제안한다

기능 단위, 리팩터링 단위, 버그 수정 단위로 의미 있는 변경이 마무리되면 커밋 시점이 왔는지 판단해야 한다.
커밋할 만한 단위가 되었다면 사용자에게 커밋을 제안하고, 원하면 바로 커밋까지 진행한다.
아직 커밋하기 이른 상태라면 남은 불안 요소나 미완성 범위를 분명히 알린다.

## 반드시. 커밋 메시지는 한글 규칙을 따른다

커밋 메시지는 한글로 작성한다.
무슨 변경을 했는지보다 왜 이 변경이 필요한지 드러나는 짧고 명확한 문장을 우선한다.
가능하면 한 커밋에는 하나의 의도를 담고, 서로 다른 목적의 변경을 묶지 않는다.

## 권장. 커밋 컨벤션은 일관되게 유지한다

커밋 메시지는 팀에서 합의한 접두어를 쓰더라도 본문 의미는 자연스러운 한글로 유지한다.
예를 들어 `feat: 예약 동시성 검증 분리`, `refactor: 결제 보상 트랜잭션 책임 이동`, `test: 좌석 점유 실패 케이스 추가`처럼 의도와 범위가 드러나야 한다.
형식보다 더 중요한 것은 커밋 단위의 응집도와 메시지의 명확성이다.
